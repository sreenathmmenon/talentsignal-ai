# Project Completion Rule

This rule applies to every task, story, feature, review, fix, demo, document, backend change, UI change, validation step, and submission artifact in this project.

## Completion Means Production-Grade

A task is not complete merely because code or documentation exists.

A task is complete only when it:

- Works correctly end to end.
- Is production-grade for the scope of the project.
- Is tested.
- Is validated.
- Handles expected edge cases.
- Has no known unresolved blocker or critical quality issue.
- Works through the backend path, UI/demo path, and exported/submitted artifact path where applicable.
- Is documented enough for another engineer, reviewer, or judge to understand and reproduce.
- Supports the goal of being world-class, first-prize-caliber, and stronger than ordinary hackathon submissions.

## Review And Fix Rule

During review, if issues are found, do not mark the work complete.

Instead:

1. Fix the issue.
2. Re-run the relevant tests, validators, demos, and end-to-end checks.
3. Review again.
4. Repeat the loop until the work is genuinely complete.

Completion requires multiple review/fix loops when needed. Do not stop at the first working draft if quality gaps remain.

## Tooling Rule

Use any appropriate tools, skills, MCP resources, scripts, AI assistants, APIs, local tools, cloud tools, notebooks, validators, profilers, UI testing tools, or manual review processes needed to reach production-grade quality.

The only exception is the official Redrob final ranking command, which must obey the challenge reproduction constraints documented elsewhere.

## Reporting Rule

Do not report a task, story, feature, or milestone as complete unless the completion standard above is met.

If something is partially done, say it is partially done and clearly list what remains.

If something works only in one path but not another, say exactly that.

If tests or validation could not be run, say that clearly and do not call it complete.
