## JD Understanding
*Free-text parsing, requirement extraction, category taxonomy, JD library.*

- **Free-text JD ingestion to weighted requirement model** — Parses ANY plain-text job description into a RequirementModel (Requirement objects + seniority band + locations) via a transparent regex + cue-lexicon pipeline, no LLM/network. — `src/talentsignal/jd_ingest.py` (ingest_text, RequirementModel, Requirement)
- **Must-have / nice-to-have / disqualifier classification** — Classifies each requirement line into must_have, nice_to_have, or disqualifier by matching section-header and inline cue phrases, with cues flipping the active section for following lines. — `src/talentsignal/jd_ingest.py` (MUST_CUES, NICE_CUES, DISQUALIFIER_CUES, _kind_for, _is_header_only)
- **Per-requirement weighting with emphasis boost** — Assigns each requirement a base weight by kind (must=1.0, nice=0.4, disqualifier=1.0) and multiplies must-haves by 1.25 when an emphasis cue is present. — `src/talentsignal/jd_ingest.py` (KIND_WEIGHT, EMPHASIS_CUES)
- **List-style requirement splitting** — Splits a bundled list sentence into individual per-skill requirements by stripping a leading cue and splitting on commas/'and'/'or'/';'/'/', guarded so prose sentences stay whole. — `src/talentsignal/jd_ingest.py` (_split_requirement_list, _LEAD_CUE)
- **Salient keyword extraction per requirement** — Extracts ordered, de-duplicated salient lowercase tokens per sentence for the lexical channel, dropping stopwords but preserving short real tech skills (go, r, c, ml, c#, .net, ci/cd) and hyphenated tool names. — `src/talentsignal/jd_ingest.py` (_keywords, _STOPWORDS, _SHORT_SKILLS)
- **Seniority band (years) extraction** — Parses experience requirements into min/max years from 'X-Y years' ranges and 'N+ years' minimums, and derives a 'strongest' inner band as the central ~60% of the preferred band. — `src/talentsignal/jd_ingest.py` (_extract_years, _YEARS_RE, _YEARS_MIN_RE); `src/talentsignal/jd_parser.py` (job_spec_from_jd_text span logic)
- **Location extraction via city gazetteer** — Detects preferred work locations by matching against a built-in Indian-city + Remote gazetteer. — `src/talentsignal/jd_ingest.py` (_extract_locations, CITY_GAZETTEER)
- **Title / intro line detection (role-agnostic)** — Identifies and skips the job-title/opening line so the role word isn't scored as a requirement, using a role-agnostic heuristic. — `src/talentsignal/jd_ingest.py` (_is_title_or_intro, _ROLE_WORD, _REQ_VERB)
- **Logistics-only line filtering** — Skips lines that are only a seniority band and/or locations so they are captured structurally rather than as scored requirements. — `src/talentsignal/jd_ingest.py` (_is_logistics_only)
- **Title auto-detection from first line** — When no title is supplied, infers the role title from the first non-empty line of the JD text. — `src/talentsignal/jd_ingest.py` (ingest_text title-guess loop)
- **Sentence segmentation + dedup** — Splits JD text into sentences on punctuation/newlines and de-duplicates repeated requirement parts by an 80-char key so the same skill isn't counted twice. — `src/talentsignal/jd_ingest.py` (_SENT_SPLIT, seen-set logic in ingest_text)
- **Requirement-model flatteners** — Helper accessors that flatten the model into must-have phrases, disqualifier phrases, and a merged keyword set, matching the JobSpec field shapes. — `src/talentsignal/jd_ingest.py` (must_have_phrases, disqualifier_phrases, all_keywords, RequirementModel.by_kind)
- **Free-text JD to scorable JobSpec** — Builds a complete scorable JobSpec directly from raw JD text — ingesting requirements, then backfilling weights, seniority defaults (5-9 yrs), and default locations from the category taxonomy where the text is silent. — `src/talentsignal/jd_parser.py` (job_spec_from_jd_text, category_profile_locations)
- **YAML scorecard loader (zero-dependency)** — Parses the project's YAML-subset job-spec files (nested maps + lists, comments, typed scalars incl. bool/int/float) without any external YAML library, and loads them into a JobSpec. — `src/talentsignal/jd_parser.py` (load_simple_yaml, _parse_yaml_with_lists, _parse_scalar, load_job_spec)
- **Unified requirement model from YAML lists** — Converts a hand-written YAML spec's must/nice/disqualifier lists into the same Requirement objects an ingested free-text JD produces, so both formats feed one scoring path. — `src/talentsignal/jd_parser.py` (_requirements_from_lists)
- **JobSpec schema with category enrichment** — Defines the immutable JobSpec structure and attaches the category's label, core signals, evidence priorities and common risks. — `src/talentsignal/jd_parser.py` (JobSpec dataclass, load_job_spec)
- **Category taxonomy with per-category profiles** — Provides six built-in role-category profiles (AI/ML search-ranking, backend, data analytics, product management, sales/GTM, design) each with label, core signals, evidence priorities, common risks, and category-tuned scoring weights. — `src/talentsignal/category_taxonomy.py` (CATEGORY_PROFILES, CategoryProfile, BASE_WEIGHTS)
- **Category fallback profile** — Returns a generic CategoryProfile for any unknown/custom category so unseen role types still resolve. — `src/talentsignal/category_taxonomy.py` (get_category_profile fallback branch)
- **Scoring-weight validation** — Validates that a spec's weight dict contains all required scoring dimensions and sums to approximately 1.0 (0.95-1.05), raising on missing keys or bad totals. — `src/talentsignal/category_taxonomy.py` (validate_weights)
- **Dual-format JD library (free-text + YAML)** — A library of six roles each providing BOTH a hiring-manager free-text JD and a structured YAML spec path, proving any-format ingestion and letting eval compare ingested-text vs hand-written scorecard. — `src/talentsignal/eval/jd_library.py` (JDEntry, JDS, AI_SEARCH_JD/SALES_JD/DATA_JD/BACKEND_JD/PRODUCT_JD/DESIGN_JD)
- **Real (non-synthetic) JD corpus** — Verbatim real public job postings (GitLab US Senior AI Engineer, India ML/recsys, India Staff Platform Engineer) with source provenance, used to validate ingestion/ranking against JDs companies actually post. — `eval/real_jds.py` (REAL_JDS)
- **Role vocabulary profiles for JD-agnostic eval** — Per-role profiles defining strong/adjacent/irrelevant titles, keyworded vs zero-shared-keyword paraphrased evidence, weak/irrelevant/honeypot evidence, skills and locations. — `src/talentsignal/eval/roles.py` (RoleProfile instances AI_SEARCH/SALES/DATA_ANALYTICS/BACKEND/PRODUCT/DESIGN, ROLES)

## Ranking Engine
*Scoring law, semantic match, rerank, ranking orchestration, evidence features, keyword baseline.*

- **Relevance-gate scoring law** — Single JD-agnostic final-score law final = clamp(R^GAMMA)*(Q_FLOOR+(1-Q_FLOOR)*Q)*soft_multiplier*hard_veto, with super-linear RELEVANCE_GAMMA=1.1 and QUALITY_FLOOR=0.45; shared by both engines via _gate(). — `src/talentsignal/scoring.py`
- **Relevance (R) vs generic-quality (Q) split** — R = how well a candidate satisfies THIS JD's parsed must-haves; Q = blend of ONLY seniority/logistics/behavioral/trust renormalized over those four (technical/career display-only). — `src/talentsignal/scoring.py`
- **Lexical spine engine (score_candidate)** — Keyword/term-coverage scoring path producing per-factor breakdown with an AI/ML-search specialized branch and a generic branch, feeding the relevance gate. — `src/talentsignal/scoring.py`
- **Hybrid embeddings+lexical engine (score_candidate_hybrid)** — JD-agnostic path where technical/career/relevance derive from the hybrid semantic match over the JD's own requirements, behavioral/trust from schema signals, penalties from the consistency auditor plus semantic disqualifier hits. — `src/talentsignal/scoring.py`, `src/talentsignal/ranking.py`
- **Whole-token graded term coverage** — _term_coverage matches requirement terms by whole tokens; multi-word terms get partial credit (0.4) when one word hits and full credit only when 2+/≥60% match; short real tech tokens preserved despite the ≥3-char filter. — `src/talentsignal/scoring.py`
- **Two-channel per-requirement semantic match** — semantic_match.match combines DENSE (rescaled cosine) and LEXICAL (whole-token overlap) as req_score = alpha*dense + (1-alpha)*lexical (DEFAULT_ALPHA=0.6), aggregated weighted by importance into semantic_fit/lexical_fit/coverage. — `src/talentsignal/semantic_match.py`
- **Rescaled cosine similarity** — cosine() computes raw cosine then monotonically stretches the informative band [0.12, 0.65] to [0,1] so true paraphrase fits separate from off-topic profiles; monotonic so it never reorders two candidates against the same requirement. — `src/talentsignal/semantic_match.py`
- **Lexical-only graceful degradation** — When embeddings are unavailable, match() falls back to req_score = lexical so the system still produces a ranking with no precomputed index or network. — `src/talentsignal/semantic_match.py`, `src/talentsignal/ranking.py`
- **Weighted must/nice semantic aggregation + coverage** — semantic_fit = 0.85*must_fit + 0.15*nice_fit (importance-weighted); coverage = fraction of must-have requirements with a real match (req_score≥0.30); both feed relevance and confidence. — `src/talentsignal/semantic_match.py`
- **Evidence spans (best_evidence_span)** — For each genuinely-matched requirement, picks the candidate's OWN sentence with the most matched keywords (ties by brevity) as the recruiter-readable 'why', truncated to 160 chars; never fabricates. — `src/talentsignal/semantic_match.py`
- **Per-requirement match records** — RequirementMatch captures req_text, kind, weight, dense, lexical, combined score, matched_keywords, and evidence_span — the grounded detail used for reasoning and drill-down. — `src/talentsignal/semantic_match.py`
- **Cross-encoder rerank / two-stage retrieval** — reranker.rerank reorders only the top_k shortlist (default 50) by scoring each (JD, candidate-evidence) PAIR with cross-encoder/ms-marco-MiniLM-L-6-v2, min-max normalizing and blending with retrieval score (default 0.5). — `src/talentsignal/reranker.py`
- **Offline-safe / never-drop reranking** — Cross-encoder loads once with HF_HUB_OFFLINE and degrades to original ranking on any failure; head items with no candidate record are re-appended so no candidate is dropped; ranks renumbered 1..N. — `src/talentsignal/reranker.py`
- **Cross-encoder explainability signal** — Each reranked candidate gets a normalized cross_encoder_score attribute surfaced for explainability alongside the blended new score. — `src/talentsignal/reranker.py`
- **Hard veto (disqualifier + impossibility)** — Score is zeroed when a semantic disqualifier requirement is hit (≥0.5) or the candidate is flagged internally-impossible by the consistency auditor; vetoed candidates become top-10 ineligible. — `src/talentsignal/scoring.py`
- **Single soft-penalty multiplier** — All non-veto penalties collapse into one soft_multiplier = clamp(1.0 - soft_penalty, 0.6, 1.0) so penalties can never exceed a 40% reduction floor. — `src/talentsignal/scoring.py`
- **AI/ML-search role specialization** — When category == 'ai_ml_search_ranking', score_candidate uses dedicated technical/career terms and applies negative career adjustments for non-tech titles and service-only profiles lacking retrieval/product evidence. — `src/talentsignal/scoring.py`, `src/talentsignal/features.py`
- **Seniority band scoring** — _seniority_score maps years-of-experience against the JD's strongest/preferred bands to 1.0 / 0.88 / 0.62 / 0.25 tiers. — `src/talentsignal/scoring.py`
- **Logistics scoring** — _logistics_score scores country match, preferred-city presence, willingness to relocate, and hybrid/flexible work mode into a clamped logistics factor. — `src/talentsignal/scoring.py`
- **Behavioral scoring** — _behavioral_score blends profile completeness, recency, open-to-work, recruiter response rate/time, notice period, interview completion, and email/phone verification. — `src/talentsignal/scoring.py`
- **Trust scoring** — _trust_score blends LinkedIn connection, GitHub activity tiers, recruiter-saves, search appearances, offer-acceptance rate, and skill-assessment max into a trust factor. — `src/talentsignal/scoring.py`
- **Schema-driven behavioral/trust blend (hybrid)** — In the hybrid engine behavioral/trust blend schema_signals (availability/engagement/trust, any schema) with the explicit-field spine scores so Redrob keeps full fidelity while arbitrary schemas still score. — `src/talentsignal/scoring.py`
- **Top matched requirements for grounded reasoning** — Hybrid path exposes the top 4 best-scoring must/nice requirements (score≥0.30) as (req_text, matched_keywords, evidence_span) tuples plus up to 3 consistency concern_notes. — `src/talentsignal/scoring.py`
- **Confidence score** — Both engines emit a confidence value tracking relevance plus evidence depth (career coverage, production terms, coverage, engagement, absence of flags). — `src/talentsignal/scoring.py`
- **Top-10 eligibility gate** — top10_eligible = not vetoed AND relevance≥0.34, a minimum-relevance bar for inclusion in the top-10 used by risk reporting. — `src/talentsignal/scoring.py`, `src/talentsignal/ranking.py`
- **Graceful degrade when JD has no must-haves** — When job.must_have is empty, relevance degrades to broad text/semantic coverage (with a 0.4 floor) so the engine still ranks rather than zeroing everyone. — `src/talentsignal/scoring.py`
- **Deterministic ranking + tiebreak** — score_pool_from_iter / score_pool_hybrid sort by (-final_score, candidate_id) for a stable order; the keyword baseline uses the identical (-score, id) tiebreak. — `src/talentsignal/ranking.py`, `src/talentsignal/baseline_ranker.py`
- **Strict non-increasing display scores** — _rows_from_scored subtracts (rank-1)*1e-6 from each display score so output scores are strictly non-increasing after rounding, preventing tie-ambiguous rank output. — `src/talentsignal/ranking.py`
- **Shared scoring path for ranking + eval** — score_pool_from_iter is shared by the file-based ranker, in-memory ranker, and the eval harness so all exercise the exact same scoring code. — `src/talentsignal/ranking.py`
- **Precomputed-index / live-embedder embedding resolution** — score_pool_hybrid resolves requirement and candidate embeddings from a precomputed index dir (or passed matrix), falls back to a live embedder callable per-candidate for demo, with lexical-only fallback when neither exists. — `src/talentsignal/ranking.py`
- **Neighbor-aware comparative reasoning hook** — _rows_from_scored passes each row's next-ranked candidate as 'neighbor' to generate_reasoning so prose can say 'edges out #N'. — `src/talentsignal/ranking.py`
- **Submission CSV writer** — write_submission emits the required candidate_id,rank,score,reasoning CSV. — `src/talentsignal/ranking.py`
- **Per-factor score export** — write_factor_scores emits a full per-candidate factor breakdown CSV (technical_evidence, career_fit, seniority, logistics, behavioral, trust, confidence, top10_eligible, penalty, risk_flags, AI-evidence term counts/terms). — `src/talentsignal/ranking.py`
- **Risk report + risk summary export** — write_risk_report emits per-candidate audit notes; write_risk_summary emits aggregate JSON (risk_flag_counts, top10_ineligible, top100_with_any_risk_flag). — `src/talentsignal/ranking.py`
- **Evidence packets (grounding audit trail)** — write_evidence_packets emits JSONL per top candidate with full score_breakdown, matched_requirements, and raw evidence text so a grounding audit can verify any cited term appears in the candidate's own profile. — `src/talentsignal/ranking.py`
- **Candidate evidence extraction** — build_evidence assembles a CandidateEvidence record from a raw candidate dict: normalized title, years, location/country, profile/career/skill text concatenations, and dozens of derived signals, tolerating skills as objects or strings. — `src/talentsignal/features.py`
- **Talent-graph term tagging** — build_evidence tags retrieval/vector/ml/eval/production terms across all-text, career-text, skill-text and title and counts product vs service company history and service-only profiles. — `src/talentsignal/features.py`
- **Title classification** — Flags non_tech_title, ai_title, adjacent_title via talent_graph title sets, used by the AI-role career adjustments. — `src/talentsignal/features.py`
- **Honeypot signal: expert-zero-duration (features)** — Counts skills claimed at 'expert' proficiency with 0 months duration and collects shallow-AI terms, surfacing self-inconsistent profile signals to downstream risk scoring. — `src/talentsignal/features.py`
- **Recency computation (months_since)** — months_since converts last-active dates to months relative to a fixed REFERENCE_DATE (2026-06-14), defaulting to 999 on missing/invalid dates. — `src/talentsignal/features.py`
- **Raw-record passthrough for auditing** — build_evidence stashes the raw candidate dict on ev._raw so downstream scoring can run the consistency auditor without re-plumbing call sites. — `src/talentsignal/features.py`, `src/talentsignal/scoring.py`
- **Keyword baseline ranker (foil)** — baseline_ranker scores every candidate by importance-weighted whole-token overlap of the JD's must/nice keywords against candidate evidence, normalized to [0,1] — an honest ATS-style keyword foil used to build the rescue ledger; read-only. — `src/talentsignal/baseline_ranker.py`

## Trust & Fairness
*Consistency/honeypot audit, risk audit, grounding audit, compliance, fairness, transparency reporting.*

- **Honeypot consistency veto (impossible-profile veto)** — Flags any HARD internal contradiction (tenure_exceeds_experience, expert_zero_duration, date_integrity, skill_exceeds_career) as is_impossible and vetoes that profile out of the top ranks; wired into both scoring paths. — `src/talentsignal/consistency_audit.py` (is_impossible, 57-61); `src/talentsignal/scoring.py` (294-298, 393); `src/talentsignal/ranking.py` (58-86)
- **Contradiction check: tenure exceeds stated experience** — Fires when summed career-history months wildly exceed stated years (beyond a generous concurrent-role overlap buffer), naming both contradicting facts; severity 0.30. — `src/talentsignal/consistency_audit.py` (125-141)
- **Contradiction check: expert proficiency with zero months used** — Flags skills marked 'expert' carrying an explicit duration_months of 0 (missing duration treated as unknown); names up to 3 skills; severity 0.15-0.25. — `src/talentsignal/consistency_audit.py` (146-156)
- **Contradiction check: single skill duration exceeds whole career** — Flags a skill whose claimed months exceed 2.5x stated total experience (impossible tail); severity 0.20. — `src/talentsignal/consistency_audit.py` (163-170)
- **Contradiction check: date integrity** — Detects end_date before start_date and start dates in the future (vs reference 2026-06) across career history; severity 0.15. — `src/talentsignal/consistency_audit.py` (172-181, _is_future/_date_before 226-239)
- **Contradiction check: claimed expert skills absent from candidate's own text (keyword-stuffing)** — Soft flag fired only when 5+ 'expert' skills exist and ALL are unfindable in the candidate's normalized evidence text; severity 0.10. — `src/talentsignal/consistency_audit.py` (194-209, _evidence_blob 78-91)
- **Contradiction check: endorsement/assessment inconsistency** — Flags a profile claiming 'expert' skills with zero endorsements received AND zero skill-assessment scores; severity 0.08. — `src/talentsignal/consistency_audit.py` (211-221)
- **Capped diminishing-returns consistency penalty** — Aggregates all consistency flag severities into a single penalty capped at 0.6; exposes flag codes and human-readable details. — `src/talentsignal/consistency_audit.py` (penalty 48-54, codes 44-46, ConsistencyFlag 33-37)
- **Malformed-input hardening of the auditor** — Auditor never crashes on messy real data: tolerates non-dict candidate, null/non-dict profile, non-list career, string-vs-dict skills, and non-finite numeric fields. — `src/talentsignal/consistency_audit.py` (94-123)

## Reproducibility & Ops
*CI quality gate, offline reproduction verifier, Docker/Render deployment, Make targets, offline precompute.*

- **Offline precompute (semantic index builder)** — Only entry point that loads sentence-transformers; embeds every candidate's evidence text and each JD requirement into pinned, L2-normalized float32 matrices persisted via artifacts.py, deterministic/offline; allowed to exceed the 5-min rank budget. — `precompute.py`
- **CI quality gate** — Single pass/fail check (exits non-zero to block merge) running on synthetic data without the 100K file, enforcing 5 guarantees so regressions can't ship silently. — `scripts/ci_gate.py` (main, MIN_COMPOSITE/MAX_HONEYPOT_AT10/MAX_CROSS_JD_OVERLAP thresholds, _load)

- **CI gate: fairness name-invariance check** — Asserts ranking score is invariant to candidate name/identity (max score delta == 0) via the fairness auditor over a labeled pool. — `scripts/ci_gate.py` (step 4); `src/talentsignal/eval/fairness.py` (audit_name_invariance)
